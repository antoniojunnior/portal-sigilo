## Problema 1

Usuário: "Estou obtendo erro ao tentar gerar os relatórios. Verifique utilizando as ferramentas do Reversa: GET https://portal-sigilo.vercel.app/api/reports/generate 500 (Internal Server Error)"

Observação do agente: relatado logo após o deploy da feature `005-relatorios-auto-geracao` (push `03f61f7` + deploy de índice Firestore, mesma sessão). A rota `GET /api/reports/generate` é chamada tanto pelo `useSWR` da página `/app/relatorios` (lista de relatórios) quanto internamente pela lógica de dedupe (`findRecentDuplicateReport`, que faz uma query diferente na mesma coleção `reports`, não pelo verbo GET do Route Handler). O erro relatado é especificamente no `GET`, não no `POST`.

Suspeita inicial (não confirmada): como `GET` e `POST` vivem no mesmo arquivo `route.ts`, uma falha de import/inicialização de módulo (ex.: `@/lib/reports/dedup`, adicionado nesta mesma sessão) derrubaria as duas rotas juntas. Precisa investigação para confirmar causa raiz — isso é trabalho do `/reversa-debugger-fix`, não deste registro.

**Hipótese mais forte (levantada nesta sessão de intake):** o `GET` faz `where("org_id", "==", ...).orderBy("gerado_em", "desc")` na coleção `reports` — isso exige um índice composto no Firestore. O `firestore.indexes.json` do repositório **nunca teve** uma entrada pra coleção `reports` antes desta sessão (só ganhou uma agora, pro dedupe do BUG-20260723-DUP1, com campos diferentes: `org_id + dedup_key + gerado_em`). Se o índice que o `GET` precisa (`org_id + gerado_em`) existia no projeto Firebase mas só via console (nunca capturado no arquivo local), o comando `firebase deploy --only firestore:indexes` rodado nesta mesma sessão pode ter **apagado** esse índice não rastreado — o CLI trata o arquivo local como fonte única de verdade e remove do projeto qualquer índice ausente dele. Isso bateria com: timing (logo após o deploy), afeta só o `GET` (a query que dependia do índice possivelmente removido), acontece sempre (índice ausente = toda query falha), e explica por que vira 500 cru: o handler `GET` não tem `try/catch` (diferente do `POST`, que captura e devolve JSON de erro) — uma exceção do Firestore (`FAILED_PRECONDITION: query requires an index`) sobe sem tratamento e vira 500 genérico do Next.js.

Confirmação real (ver índices existentes no console Firebase, ler o log da função na Vercel) é trabalho do `/reversa-debugger-fix`, não deste registro.
