Plano de correção · Checkout/renovação enviam payload errado à Asaas

BUG-20260721-K9M2 (#1) + BUG-20260721-V3F7 (#7) · contexto unificacao-plano-assinatura · gerado 2026-07-21
critical P0

Defeito e causa raiz

K9M2: src/lib/asaas/createPaymentLink.ts envia installmentCount/totalValue a POST /v3/paymentLinks. V3F7: functions/src/renovarAssinatura.ts envia value (sem installmentValue) a POST /v3/payments. Os dois foram confirmados (não hipótese) contra o sandbox Asaas real nesta sessão: os payloads atuais retornam 400 sempre; os payloads corrigidos retornam 200.

// atual (K9M2) → 400 invalid_object
POST /v3/paymentLinks { chargeType: "INSTALLMENT", installmentCount, totalValue }

// corrigido → 200
POST /v3/paymentLinks { chargeType: "INSTALLMENT", maxInstallmentCount, value }

// atual (V3F7) → 400 invalid_installmentValue
POST /v3/payments { value, installmentCount, creditCard, ... }

// corrigido → 200, creditCardToken presente na resposta
POST /v3/payments { installmentValue, installmentCount, creditCard, ... }

Evidência completa: evidence/reproduction.md (4 chamadas reais ao sandbox, request+response).

Estratégia escolhida

Correção direta. Causa raiz já confirmada com evidência real de sandbox, sem hipóteses concorrentes — debate multiagente não agregaria aqui.

Testes planejados

  • Reprodução: já capturada em evidence/reproduction.md — as chamadas reais aos payloads atuais, mostrando o 400 (não será re-executado o payload quebrado propositalmente após o fix; a evidência congelada já prova o "antes").
  • Regressão (CHG-003): scripts/test-asaas-billing-payloads.ts chama o sandbox real com os payloads corrigidos e assevera: (a) POST /v3/paymentLinks → 200 + url presente; (b) POST /v3/payments com cartão de teste → 200 + status: "CONFIRMED" + creditCard.creditCardToken presente. Roda contra sandbox real (não mock), então protege contra o nome de campo errado voltar E contra a Asaas mudar o contrato no futuro.

Riscos

Risco: média

  • Blast radius pequeno: 2 arquivos, troca de nomes de campo, sem migração de dado.
  • Contrato externo financeiro: qualquer engano aqui afeta cobrança real de dinheiro — por isso o teste roda contra sandbox real, não mock.
  • Fora desta correção (Agent Notes dos bugs): a confirmação de que o creditCardToken chega de fato no webhook PAYMENT_CONFIRMED após um pagamento completado através da URL do Payment Link (não testável via curl, precisa de navegador simulando o checkout hospedado da Asaas) segue como root_cause.state: confirmed só para o erro 400; a arquitetura de tokenização ponta-a-ponta via link permanece supported, não confirmed. Recomendo um teste manual real (T025 do roadmap) antes de considerar a feature pronta para produção.
  • Arredondamento de centavos em installmentValue (CHG-002): a Asaas pode validar que installmentValue × installmentCount bate com o valor total dentro de alguma tolerância — o script de teste (CHG-003) vai expor isso se for o caso.
