Teste de concorrência real

Situação: mitigação já aplicada no código, falta formalizar e testar de verdade

A transação Firestore (runTransaction) já está no código (marcador FIX BUG-11), mas está inline dentro do Route Handler — não há como escrever um teste automatizado real de concorrência sem invocar o handler HTTP inteiro (sessão/cookie). Esta correção extrai a lógica de reserva de slot para uma função testável e escreve um teste que dispara 2 chamadas verdadeiramente concorrentes contra o Firestore real, provando que só uma passa.

Resumo do defeito e causa raiz

confirmed Antes do fix: o endpoint lia gerado_em, chamava a Anthropic API (pode levar segundos), e só DEPOIS gravava o novo gerado_em. Duas requisições concorrentes liam o mesmo valor antigo antes de qualquer uma escrever — ambas passavam pelo rate limit de 24h.

Estratégia escolhida

Correção direta (mitigação já presente no código via runTransaction). Formalização: extraio a transação pra reserveRegenerationSlot(orgId) em src/lib/insights/rateLimit.ts — mesmo padrão de extração-para-testabilidade já usado em mapItems.ts/BUG-T6R2 nesta sessão — e escrevo um teste real de concorrência (Promise.all com 2 chamadas simultâneas contra Firestore real).

Testes planejados

TipoO que prova
Regressão (concorrência real)Org de teste sem gerado_em prévio; 2 chamadas a reserveRegenerationSlot disparadas via Promise.all — exatamente 1 retorna allowed: true, a outra allowed: false. Prova que a transação serializa corretamente, sem mockar o Firestore.

Riscos

Risco: baixa-média. Extrair a transação pra função separada é refactor pequeno e cirúrgico — mesma lógica, só movida de lugar. Risco real é de teste "flaky": concorrência real em CI pode ter timing variável; mitigado testando contra Firestore real (não emulador, que pode ter latência de rede diferente) e checando a invariante (exatamente 1 allowed=true), não o timing exato.
Fora do escopo (já registrado no bug): se a chamada à Anthropic API falhar DEPOIS da transação reservar o slot, a regeneração é perdida (usuário só pode tentar de novo em 24h). Trade-off aceito e documentado — não é um bug novo, é a consequência conhecida da escolha "reservar antes de chamar a API".
