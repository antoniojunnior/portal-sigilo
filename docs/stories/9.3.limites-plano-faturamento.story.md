## Story

**As a** gestor admin de uma org no Portal Sigilo,
**I want** que os limites do meu plano sejam aplicados automaticamente e que eu possa ver meu plano e data de renovação no dashboard,
**so that** eu tenha visibilidade do meu plano contratado e o sistema previna uso além dos limites de forma transparente.

## Acceptance Criteria

1. **Limite de usuários gestores por plano — Firestore Rules**
   - Plano Entrada: máximo 1 usuário gestor (role: admin | gestor | auditor) por org
   - Plano Gestão: máximo 10 usuários por org
   - Plano Enterprise: ilimitado
   - Firestore Rule bloqueia `create` em `users` se contagem >= limite do plano
   - Regra aplicada via `getAfter()` ou contagem em campo `users_count` na org

2. **Limite de armazenamento — Firebase Function**
   - Antes de aceitar upload em Storage, Firebase Function verifica storage usado pela org
   - Plano Entrada: 2 GB; Plano Gestão: 20 GB; Enterprise: sem limite
   - Retornar 403 com `{ error: 'storage_limit_exceeded', used, limit }` se excedido
   - A verificação ocorre na Function de upload já existente no projeto

3. **Orgs suspensas e canceladas**
   - Org com `plano_ativo: 'suspenso'` ou `'cancelado'`:
     - Dashboard exibe banner de aviso persistente no topo com link para regularizar
     - Funcionalidades de geração de relatório e assistente IA ficam bloqueadas (retornam 403)
     - Login ainda funciona — gestor precisa ver o aviso
   - Verificação feita no middleware ou em cada Route Handler relevante

4. **Tela de faturamento no dashboard (`/app/configuracoes/faturamento`)**
   - Exibe: plano atual (badge colorido), data de renovação formatada, status da assinatura
   - Botão "Gerenciar assinatura" → Route Handler `/api/billing/portal` retorna URL do portal do cliente Asaas
   - URL do portal Asaas gerada via API Asaas server-side, nunca hardcoded
   - Seção visível apenas para usuários com `role: admin`

5. **Badge de plano no header do dashboard**
   - Badge exibido no header do dashboard com o plano atual: "Entrada" | "Gestão" | "Enterprise" | "Suspenso"
   - Cor: cinza (Entrada), azul (Gestão), roxo (Enterprise), vermelho (Suspenso/Cancelado)
   - Badge clicável — direciona para `/app/configuracoes/faturamento`

6. **Compatibilidade**
   - `plano_ativo` já verificado nas Fases 4, 5 e 6 — sem breaking changes
   - Orgs de teste existentes com `plano_ativo: 'gestao'` continuam funcionando

### Firestore Rules — padrão existente no projeto

```javascript
// Padrão atual (verificar firestore.rules para sintaxe exata do projeto)
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Helper para limite de plano
    function getPlanoLimit() {
      let plano = get(/databases/$(database)/documents/orgs/$(request.auth.token.org_id)).data.plano_ativo;
      return plano == 'entrada' ? 1 : (plano == 'gestao' ? 10 : null);
    }

    match /users/{userId} {
      allow create: if isAdmin()
        && (getPlanoLimit() == null
            || resource == null  // primeira criação
            || get(/databases/$(database)/documents/orgs/$(request.auth.token.org_id)).data.users_count < getPlanoLimit());
    }
  }
}
```

> **Nota:** Usar `users_count` como campo denormalizado em `orgs` é mais eficiente que `getAfter()` count query em Firestore Rules. Manter sincronizado via Firebase Functions ao criar/desativar usuários.
